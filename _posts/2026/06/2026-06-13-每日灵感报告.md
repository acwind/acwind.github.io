## 1. Putt.day — 每日迷你高尔夫

**核心功能**：每天推出一张新的迷你高尔夫球洞地图，玩家通过网页即可快速击球，主打"每日一洞"的轻量社交竞技体验。

**创意来源**：HN Show 上由 ellg 展示的项目，短短一天收获 136 个点赞和 67 条评论，热度远超同期其他项目。这种"每日挑战 + 排行榜 + 极简操作"的组合验证了用户对轻量化每日游戏的需求。（来源：https://putt.day/）

**技术路径**：前端用 Vite + vanilla TypeScript 或轻量 React 即可，物理引擎推荐 Matter.js 做球的碰撞和反弹模拟，排行榜用 Supabase 提供的实时订阅能力，地图数据用 JSON 静态托管。不需要后端，纯静态站点加 BaaS 即可支撑。

**差异化优势**：相比传统高尔夫游戏动辄几百 MB、需要下载客户端、规则复杂，Putt.day 浏览器即开即玩，一局不到一分钟。将"每日一图"的 Wordle 式运营思路嫁接到物理小游戏上，用日报机制制造每日回访的钩子，比单纯做一个高尔夫模拟器聪明得多。

## 2. MarkMark — Markdown 批注工具

**核心功能**：在 Markdown 渲染界面上直接圈点、划线、添加批注标注，并导出为标准 CriticMarkup 格式，让文档审阅回归纯文本工作流。

**创意来源**：V2EX 用户 easychen 分享道："现在编码时有一半的时间都是在做审批。但是绝大部分的 Markdown 阅读器都不支持添加标注，所以我做了一个 MarkMark。"这个痛点精准击中了程序员和技术写作者的日常——PR 审查、文档协作、博客校对都需要标注能力，但现有工具要么绑定特定平台（如 GitHub PR review），要么需要切换格式（如导出 Word 再批注）。（来源：https://github.com/easychen/markmark）

**技术路径**：推荐用 React 框架，核心依赖 markdown-it 做 Markdown 解析渲染，CriticMarkup 的插入/删除/高亮/注释四种标记直接在渲染后的 DOM 层做选区映射。标注数据用 localforage 存本地 IndexedDB，支持导出为 CriticMarkup 文本和带标注的 HTML。

**差异化优势**：Notion、HackMD 等协作工具虽然功能强大，但标注都绑定其自有格式，无法拿到本地 Markdown 文件用 Git 跟踪变更。MarkMark 的对标是"Markdown 世界的 PDF 批注工具"——不改动原文，标注信息以标准格式存储，可以和任何文本编辑器、Git 工作流无缝衔接。功能单一但精准，一周可出 MVP。

## 3. Babel — 实时 AI 语音翻译通话

**核心功能**：匹配两位说不同语言的陌生人进行匿名语音通话，AI 在通话中实时翻译双方的语音，实现"各说各话、互相听懂"的实时跨语言对话。

**创意来源**：HN Show 上 epsteingpt 展示的项目。作者称 85% 的代码是用前沿模型一次生成的，实时 WebRTC 部分调试花了一些功夫。这个创意将目前成熟的语音识别、翻译、TTS 三条 AI 管线组合成一个有温度的社交产品，而不只是又一个 AI 翻译工具。"和陌生人聊天"的随机匹配机制本身就带有社交产品的传播基因。（来源：https://itsbabel.com）

**技术路径**：前端用 Next.js + WebRTC（Simple-Peer 或原生 RTCPeerConnection），语音转文字用 OpenAI Whisper API，翻译用 GPT-4o-mini 或 DeepL API，文字转语音用 ElevenLabs 或 Azure TTS。后端用 Node.js + Socket.IO 做信令服务器和匹配队列。关键难点在于延迟控制——需要将端到端延迟控制在 2 秒以内对话体验才自然。

**差异化优势**：相比谷歌翻译的对话模式（双方共用一台手机轮流传声），Babel 是真正的"戴着耳机打电话"体验，更像同声传译。它没有把翻译做成工具属性，而是做成了社交属性——匹配陌生人这个机制本身就创造了使用理由。功能精简（只有翻译通话），不搞社区不搞好友系统，单点突破清晰。

## 4. 保价精灵 — 电商自动保价插件

**核心功能**：Chrome 浏览器插件，在后台自动监控已购商品的当前价格，发现降价时自动申请价格保护退款，用户不需要每天手动登录京东检查。

**创意来源**：V2EX 用户 nnlsm 分享了他做的 Chrome 插件"保价精灵"。京东保价政策在商品降价时可退差价，但需要用户自行发现并手动申请——大多数人根本不会每天去检查。插件每天自动执行这一操作，免费版每月 1-15 号保价，付费版天天保价。（来源：https://gitee.com/nnlsm/baojiajingling）

**技术路径**：Chrome Extension Manifest V3，background service worker 定时触发，用 Puppeteer 或直接调京东内部 API 抓取已购商品当前价格，对比订单价格后通过京东保价接口自动提交申请。Puppeteer 太重的话可以考虑直接分析京东的 HTTP 请求链，用 fetch 模拟。数据用 chrome.storage.local 持久化。

**差异化优势**："自动保价"这个需求极其明确——帮用户省下本可以拿回来的钱。它不做购物比价（已有太多），不搞优惠券推送（用户反感），只做一件事：监控已买商品、自动退差价。单一功能，零学习成本，装了就能省钱。并且这个模式可以横向扩展到淘宝、拼多多等其他电商平台，形成"全平台保价"的差异化壁垒。

## 5. TransitWake — 通勤智能闹钟

**核心功能**：闹钟 App 在早上自动查询实时公交/地铁到站时间，如果通勤线路发生延误，自动提前叫醒用户，避免因交通延误而迟到。

**创意来源**：Reddit r/SomebodyMakeThis 上韩国开发者 OkTwist4996 提出的想法："我正在认真考虑做一个闹钟 App，早上自动查实时交通班次，如果有延迟就提前叫醒你。"评论区反响热烈，多个用户表达了"绝对会用"的强烈意愿，尤其在大城市通勤族中。（来源：https://www.reddit.com/r/SomebodyMakeThis/comments/1u3o9nl/）

**技术路径**：iOS 端用 SwiftUI 开发原生 App（闹钟权限需要原生能力），公共交通数据接入各城市的 GTFS 实时数据（Google Transit APIs 或各城市开放数据 API）。核心逻辑很简单：用户设定"期望到达时间"和通勤路线，App 在闹钟触发前 10 分钟查询实时班次状态，若预计到达时间晚于用户设定的截止时间，则提前触发闹钟。后端可用 Cloudflare Workers 做 API 代理层。

**差异化优势**：市面上闹钟 App 千篇一律，但没有任何一个将"交通延误"这个变量纳入唤醒决策。它本质上是把"实时交通数据 + 闹钟"这两个已有功能做了一次有意义的交叉——不需要重新发明闹钟，也不需要重新发明交通查询，但把它们连接起来就创造了一个全新的使用场景。功能极其纯粹，MVP 一周可完成。
